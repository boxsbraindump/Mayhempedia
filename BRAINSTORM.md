# 海克斯大乱斗 Overlay —— 设计文档 (BRAINSTORM)

> 一个英雄联盟「海克斯大乱斗 / ARAM: Mayhem」的桌面 overlay 软件。
> 双重目标：**① 自己日常玩大乱斗真用得上 ② 作品集亮点项目**。

---

## 0. 一句话定位

**「大乱斗流派副官」（ARAM Build Copilot）** —— 进对局前自动认出你被随机到的英雄、让你选一个**出装流派**（如 AP暴击卡莎 / AD攻速卡莎）；对局内快捷键呼出 overlay，实时陪你**出装 + 选海克斯增强**。

**产品聚焦成一句话：进游戏前选好你这把的流派，局内 overlay 全程陪你出装、选增强。**

> ⚠️ 已砍掉的方向：**「选人副官」（板凳/reroll 建议）** —— 大乱斗的核心体验是"随机给你个英雄、你把它玩明白"，没人因为胜率高才选英雄，这是伪需求。（2026-06-28 决策）

---

## 1. 为什么做这个（机会缺口）

**最接近的竞品 = 海斗小助手**（微信小程序，见截图）。它已经把"英雄页 → 多流派出装 + 每流派的核心/备选海克斯 + 陷阱标注"的**内容层做得很完整**（748 构建方案 / 173 英雄）。

**但它是"死的"：**
- 微信**小程序 = 被动查询工具**，你得手动搜英雄、盯着手机/另一块屏。
- 它**不能**盖在游戏上、**不能**自动感知你被随机到谁、**不能**在你选增强那一刻给你框出来。

**我们的差异化 = 同样的内容，用"活的实时 overlay"送到眼前。**
海斗小助手证明了需求真实、内容模型可行；而"**实时、自动、就在你出装/选增强的那一刻**"正是小程序永远做不到的——这就是护城河。参考形态 = TFT 的 overlay（MetaTFT / TFT Academy：自动识别对局、局内框出推荐）。

**市场趋势佐证**：老三家（Porofessor/Blitz/Mobalytics）都依赖 Overwolf，正被「无 Overwolf + 轻量」新势力（iTero 等）蚕食 → 「独立、轻量、专精」是被认可的方向 = 我们的路线 B。

## 3. 技术路线决策

**路线 B：独立桌面 app（Electron + TypeScript, ESM），不走 Overwolf。**（已定，2026-06-28）

- 理由：作品集含金量（自己处理实时数据/本地认证/透明窗口/CV）；契合「无 Overwolf 轻量」趋势；日常自用足够。
- Electron 起步：`league-connect` 是 Node 一等公民、透明窗口生态成熟。难点应放在"实时数据 + 流派逻辑 + CV"，而非框架本身。Tauri 留作后续"内存优化"迁移故事。
- 前端：React + TypeScript。

### ⚠️ 合规边界（红线，决定能不能日常用）

只用官方/容忍的**只读**接口，绝不读内存、不自动操作、不注入进程（否则封号）：

| 接口 | 用途 | 性质 |
|---|---|---|
| **LCU API** (`https://127.0.0.1:<port>`, 认证靠 `lockfile`) | **自动识别你被随机到的英雄**、锁定状态、召唤师信息 | 非官方但长期被容忍 |
| **Live Client Data API** (`https://127.0.0.1:2999/liveclientdata/...`) | 对局内：游戏时间/等级/金币/装备（用来判断"何时该发增强了"） | Riot 官方文档支持 |
| **Community Dragon** | 静态资源：英雄/装备/符文 + **海克斯增强图标 & 数据** | 社区维护，比 Data Dragon 更全 |
| **屏幕图像识别（V2）** | 认出局内被塞的 3 个增强（API 拿不到，见 §5.5） | 只读自己屏幕像素 → 合法 |

**❗合规发现（2026-06-28，见 Riot 开发者政策）**：产品**不能显示海克斯增强/斗魂竞技场物品的"胜率"数字**，也不能提供"玩家本来不知道的对局专属信息"。
→ 设计对策：用**定性标签（核心 / 备选 / 陷阱）替代裸胜率**（正好是海斗小助手的做法）；只标注**玩家自己屏幕上看得到的东西**（自己的三选一），不碰隐藏/对手信息。这跟我们"流派 curated 优先级"设计天然一致。

## 4. 架构（路线 B）

```
┌─────────────────────────────────────────────┐
│  Electron 主进程 (Node)                        │
│  ├─ LCU 连接器     → 自动识别当前英雄/锁定状态    │
│  ├─ Live Client 轮询 → 游戏时间/等级(判断发增强点) │
│  ├─ 流派数据       → 手工 curate 的 JSON         │
│  ├─ 本地数据快照    → data/*.json + icons (见下)   │
│  └─ 【V2】CV 识别器 → 截屏+图标哈希认出3个增强     │
└───────────────┬─────────────────────────────┘
                │ IPC
┌───────────────▼─────────────────────────────┐
│  透明置顶 overlay 窗口 (React) —— 快捷键呼出     │
│  左上角长方形：出装顺序 + 增强优先级(标签)        │
└─────────────────────────────────────────────┘
```

**透明置顶 overlay 窗口**（Electron 关键配置）：
```js
new BrowserWindow({ transparent:true, frame:false, alwaysOnTop:true, skipTaskbar:true })
win.setAlwaysOnTop(true, 'screen-saver')          // 层级够高
win.setIgnoreMouseEvents(true, { forward:true })  // 点击穿透到游戏
```
⚠️ **全屏坑**：盖得住**无边框窗口模式**，盖不住独占全屏。文档提醒用户切无边框（绝大多数 overlay 用户都这么玩）。

### 数据存储决策：本地快照（已定 2026-06-28）
**运行时纯本地，联网只发生在"更新数据"这一个动作里。**
```
本地 bundle 快照 (data/*.json + icons)  ← 运行时只读这个，快而稳
        ▲ npm run data（打补丁时手动/以后自动跑）
Community Dragon (latest)              ← 只在更新时碰网，绝不在对局中碰
```
理由：① overlay 在对局中读远程图标 = 网络一抖就闪烁/白屏，本地无此风险；② CDragon 路径会变（已踩坑：增强图标在 `game` 分支、装备在 `plugins` 分支）；③ **V2 的 CV 图标哈希库本来就必须用本地图标**，既然图标必须落地就统一本地；④ 断网也能 demo。代价（数据过期）用 `npm run data` 重跑解决；进阶做启动时比版本号自动更新。

## 4.5 性能预算（"不卡"原则）

> 上线后最怕"用户卡"。结论：**MVP 不会卡；唯一真正的风险是 V2 的 CV。** 把约束提前钉死，别写飞。

**卡不是来自数据存储**（读本地 JSON+图标是毫秒级）；来自两处：

1. **Electron 基础占用**（~150–250MB 内存）：只影响弱机，且**静止的 overlay 几乎不占 CPU/GPU**，不掉帧。MVP 安全。兜底杠杆 = 若占用成为抱怨点，迁 Tauri（内存砍到 ~10–30MB）。
2. **V2 CV 截屏识别 = 唯一要小心的**。连续截屏+图像处理才吃 CPU/GPU。铁律（做 CV 时必须遵守）：
   - **事件门控**：只在"快发增强了"（游戏时间/等级判断）的窗口激活，平时**根本不跑**。
   - **只截一小块**：增强选择区域，不截全屏。
   - **低频**：界面弹出时抓几帧即可，不是每帧。
   - **后台线程**：worker/native 跑，不占渲染主线程。
   - pHash 匹配小区域是毫秒级——重的是"一直截屏"，而我们根本不一直截。

**渲染侧通则**：overlay 默认隐藏、快捷键呼出；隐藏时不渲染；静态内容极少重绘；避免重动画/backdrop-blur；点击穿透不抢输入。目标：idle/静态可见时对游戏 FPS 近乎零影响。

### M3.1 —— ✅ 已完成（2026-07-03）：真机测试反馈修复

- **"海克斯长得像，不给名字根本看不清"**：图标列表原来只有 hover 时的 `title` 提示——但 overlay 默认点击穿透，鼠标事件根本不落在窗口上，hover **永远不会触发**，等于名字提示是摆设。改成名字文字常显（图标+文字并排），不再依赖 hover。
- **"位置固定不能拖，怕挡视线"**：新增**解锁/锁定拖动模式**（默认 `Ctrl+Shift+L`，可在设置页看到），效果参照 TFT 类插件——按一下解锁：关掉 `setIgnoreMouseEvents`，面板变红色虚线框+`-webkit-app-region:drag`，可以像普通窗口一样拖动；再按一下锁定：恢复点击穿透，把当前坐标存进 `settings.overlay.customPos`（新字段，覆盖原来的四角锚点算法）。设置页加了"已手动定位到(x,y) + 重置为锚点默认位置"入口。
- 顺带再次印证了 V2 CV 方案的必要性：文字常显只解决"认出已经会给你看的清单里的名字"，解决不了"游戏本身弹出的 3 选 1 认不出是哪个"，这个仍然只能靠下面的 V2。
- **验证边界**：预览环境里用临时内存态分别验证过锁定/解锁两种视觉状态(边框/光标/文案切换正常)、设置页新增的 UI 都能正确读写。**真实拖动手感、穿透游戏点击的实际体验**仍需用户在真机确认。

**功能设想**：

- 开放用户自己上传流派（不再局限于我们curate的173个英雄库，变成UGC内容池）。
- 给网页端（如果做的话）加评论区+打分，形成"哪套流派好不好"的社区共识信号。

## 5.5 核心机制设计

### 为什么"动态推荐下一个增强"不做（双重否决）
需求原型曾是"给定增强1+装备，推增强2"。否决，两个独立原因：
1. **数据稀疏**：按 `英雄×增强1×增强2×装备` 切片样本太少，噪音盖过信号；竞品也只发一维胜率。
2. **合规**：Riot 政策不让显示增强胜率数字。

### 关键反转：增强是「三选一」，不是「自由搭配」
系统随机塞 N 个（通常 3）、你选 1。真需求 = **"眼前这 3 个哪个最配我这把流派？"** = 「按流派优先级给三选一排序」。只用**一维数据 + 流派人工权重 + 定性标签**，可解、合规、不撞竞品。

### 流派（Archetype）数据模型
```
champion → archetypes[] → {
  name,                       // "AP暴击流卡莎"
  items[],                    // 出装顺序
  runes,                      // 符文
  augmentTiers: {             // 该流派下增强的定性分级（不放胜率数字）
    core:   [augmentId...],   // 核心（框住）
    good:   [augmentId...],   // 备选
    trap:   [augmentId...],   // 陷阱（避免）
  }
}
```
- 先**手工 curate 自己常玩的英雄**（自用起步，逐步扩），可用海斗小助手/aramgg 的公开分级做参考种子。
- MVP：静态展示清单；V2：CV 认出屏上 3 个 → 查 `augmentTiers` → 框出 core。

### V2 CV 难度评估（2026-06-28）
局内那 3 个增强 **API 拿不到**（Live Client Data 不暴露），必须**截屏 + 图像识别**（合法，只读自己屏幕；TFT overlay 都这么做）。拆 3 块：

| 子问题 | 方案 | 难度 |
|---|---|---|
| ① 认出 3 个增强是啥 | 裁图标区 → 跟 Community Dragon 图标库做 **pHash/模板匹配** | 🟢 低（图标固定，比 OCR 中文名稳） |
| ② 何时触发识别 | 用 Live Client 游戏时间/等级判断"快发增强了"→ 只在窗口内激活；或视觉检测增强界面 | 🟡 中 |
| ③ 分辨率/UI缩放适配 | 按分辨率标定坐标，或先检测增强卡区域 | 🟠 中偏高（最烦最脆，吃时间） |

- 结论：**可行、合法**，参考 TFT 的 tft_augments_helper（一人用 Tesseract 做出）。真正难点是 ②③ 工程脏活，不是 ①。
- 工作量：能用版本约 **1~2 周 + 持续调优**；脆（改版/新分辨率要维护）——但正因难又脆，**是作品集最亮的技术故事**。
- **提前量（低风险、不阻塞 MVP）**：可先建"**增强图标 pHash 数据库**"（拉 Community Dragon 全部增强图标→生成哈希），是 ① 的地基，单独可验证。
